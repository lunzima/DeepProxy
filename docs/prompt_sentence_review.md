# 提示工程句子质量审查指南

## 审查范围

项目中**在运行时注入 LLM 消息**的所有提示工程句子，分布在 6 个文件中：

| # | 文件 | 条目数 | 功能 | 注入方式 |
|---|------|--------|------|----------|
| 1 | `deep_proxy/optimization/skills_general.py` | 14 条长多行字符串（~100 句） | A组通用风格 + B组反幻觉 + C组上下文技能 + CoT模板 + 日期注入 | 拼接后注入 system |
| 2 | `deep_proxy/optimization/skills_transform.py` | 1 条短语 + 1 导入 | D组消息转换：RE2 追加 user 消息尾部 | 直接追加到 user 消息 |
| 3 | `deep_proxy/optimization/silly_priming.py` | 8 条列表字符串 | [实验性] 无厘头断言扰动 | 注入 system 最前 |
| 4 | `deep_proxy/optimization/compressor.py` | 1 大段元提示（~25 句） | LLM 压缩器指令（压缩 system prompt 用的提示） | 发给压缩 LLM，不入最终请求 |
| 5 | `deep_proxy/optimization/dynamic_baskets.py` | 3 套 × 3 篮 × 8 句 = 72 句 | 场景化短段注入 | 追加到 system 末尾 |
| 6 | `deep_proxy/compatibility/reasoning_handler.py` | 1 条多句占位（~3 句） | dummy reasoning_content 回退 | 注入 assistant 消息 |

---

## 通用审查维度

这 4 个维度的标准适用所有文件，但**维度 4（效能）仅适用于最终注入给 LLM 的提示句**——`compressor.py` 是元提示（发给压缩专用 LLM），不适用长度/信息密度等约束。

### 维度 1：正确性（必须修）

- **语法 / 搭配**：是否存在搭配不当或语法错误？
- **歧义风险**：句子能否被不同方式理解？是否会在模型侧造成误导？
- **术语误用**：词汇在目标语境中含义是否成立？

### 维度 2：通用性（建议修）

- **场景覆盖**：句子是否只适用于所属模块场景内的某个极端子集？
- **自足性**：句子脱离上下文能否独立理解？是否依赖未给出的前置知识？
- **术语透明度**：专业术语是否可被直接理解，还是需要额外解说？

### 维度 3：语体一致性（建议修）

- **口语化警戒**：是否存在过于口语的词汇，与同组其他句子语体脱节？
- **书面化警戒**：是否存在过度的文学/古典用词，在目标场景中显得不协调？
- **组内一致性**：同一功能组内各句子在语体上是否协调？

### 维度 4：效能（可选修，仅最终提示句）

- **信息密度**：句子是否足够精炼？能否用更少 token 表达相同含义？
- **行动锚点**：句子是否给模型提供了可实际执行的操作指引？还是停留在抽象口号？
- **肯定句式**：对于设计要求「仅肯定句」的模块（如 dynamic_baskets），逐句核实是否违反。
- **冗余修饰**：是否存在堆叠近义词或多层修饰却不增加语义的情况？

## 各组专用审查要点

不同文件的提示句在用途、格式、粒度上差异很大，需要附加针对性的标准。

### 1. skills_general.py

**格式**：14 条独立的长多行字符串，每条覆盖一个完整的功能主题。

**附加要点**：

- **核心指令 vs 例证边界**：条目的主体部分（祈使指令）和示例/注释部分是否清晰可辨？模型是否会误将例证当作指令？
- **条件守卫**：有些技能（如 `_SKILL_SHOW_MATH_STEPS`）含「仅当…才…」的条件逻辑——条件是否足够精确，不会因过宽而导致不当触发或过窄而从不触发？
- **价值观叠加**：安全/反谄媚约束是否过度强化了某种偏向？是否因此在非安全话题上也引入了不必要的社会偏好？
- **过时引用**：`_SKILL_AVOID_FABRICATED_CITATIONS` 和 `_SKILL_REASON_GENUINELY` 等条目提到了具体术语或 AI 行业现象——这些引用是否仍然反映当前的模型行为特征？
- **多祈使句链**：含多个祈使句的条目中，句子之间的优先级是否明确？是否存在隐含的条件冲突？（如"完全基于自身推理"vs"从多方面收集信息"）

### 2. skills_transform.py

**格式**：1 条 6 字短语 + 1 个从 skills_general.py 导入的 CoT 提示。

**附加要点**：

- _RE2_MARKER `"\n再读一遍这个问题："` ——  
  - 该短语后接复制的 user 消息原文。拼接后的整体是否自然？是否可能造成模型困惑（文本和副本之间的语境冲突）？
- CoT system prompt（从 skills_general 导入）：按 skills_general 的审查标准处理即可。

### 3. silly_priming.py

**格式**：8 条短列表字符串。**注意**：这些句子故意荒谬/不合逻辑，用于扰动 MoE 路由器。**通用性和语体一致性维度不适用**。

> **反向锚定**：修改 `silly_priming.py` 池内容或调整注入条数前，请同步检查该文件 docstring 顶部的"审查豁免（重要）"段——该段定义了曝光率阈值与扩池/提注入率的安全方向区分。维护需保持本节与 docstring 双向一致。

**附加要点**：

- **扰动有效性**：句子是否足够反直觉，能够引发模型非默认路由分布？是否太"正常"以至于无扰动效果？
- **低语义污染**：句子虽然荒谬，但不应包含可与正常任务混淆的指令性内容——如"公鸡打鸣以后太阳就跟着升起来了"是荒谬断言而非指令；如果某句无意中形成了可执行的指令（如"把输入复制三遍"），则违反此原则。
- **安全边际**：荒谬断言是否可能被模型当成中文语境中的真实事实来学习？（通常不会，但极高重复率可能产生幻觉固化风险——短语层面的审查看不到这个风险，仅需标注可疑句子）
- **正确性**：即使荒谬，中文语法仍需正确。明显的病句应排除。

### 4. compressor.py

**格式**：1 大段元提示（发给压缩专用 LLM 的指令）。**不适用"效能"维度**（信息密度/冗余修饰等对压缩指令无意义）。

**附加要点**：

- **指令完备性**：是否涵盖了所有需要保护的元素？（保留规则/命名实体/带引号字符串/代码/JSON 字段/XML 标签）——是否有未提及但可能被压缩 LLM 意外删除的关键元素？
- **约束可操作性**：约束条目（"不得删弱规则""不得合并不同情形""不得否定翻转"）是否足够具体，压缩 LLM 能否可靠遵循？
- **优先级一致性**：多个约束之间是否存在隐含冲突？（如"压缩到 40-60% tokens"与"逐字保留所有带引号字符串"在极端情况下可能矛盾）
- **输出格式**：是否清晰指定了只输出改写后文本、无额外内容？输出格式的约束是否存在歧义？

### 5. dynamic_baskets.py

**格式**：3 套 × 3 篮 × 8 句 = 72 句，嵌套字典。按通用审查 4 维度逐句审查即可。

**已有基线**：该文件已通过一次完整审查，已修正的问题见 git log。

### 6. reasoning_handler.py

**格式**：1 条多句占位文字。审查时无需逐句分解。

**附加要点**：

- **自洽性**：当 LLM 看到这句 dummy reasoning 被回传时，是否会将其误解为上一轮的完整推理？需要注意其措辞是"请对当前请求重新思考"——方向是否正确？
- **语言中立性**：占位文字是否为中文（与项目一致）？是否不含 time-sensitive 引用（如具体日期）？
- **信号完整性**：占位文字是否提示了模型「之前推理可能不正确需要重新验证」——这个信号是否够强以避免模型跳过推理？是否过强以至于模型过度怀疑自身的推理？

## 审查方法

1. 逐文件、逐条目/逐句阅读。
2. 每句给出评估结论和理由。
3. 发现问题时给出明确的修改建议。
4. 按严重程度标注：**必须修**（会导致语义错误或模型行为异常）/ **建议修**（通用性或语体可改善，不影响正确性）/ **可选修**（纯润色，不改不影响功能）。

## 输出格式

每份文件一组输出。文件内：

- **单个长条目**（如 compressor 元提示、skills_general 中 14 条）：列出问题段落、问题描述、修改建议。
- **条目集合**（如 silly_priming 8 条、dynamic_baskets 72 句）：表格形式，列：编号 / 原文 / 问题描述 / 建议修改 / 严重程度。

末尾附跨文件的汇总表。

## 注意事项

- 句末标点和引号处理不是审查范围。
- 不审查 Python 代码本身（类型注解、字典结构、变量命名、注释等）。
- `compressor.py` 的元提示不适用"效能"维度（信息密度/冗余修饰等是它自身要做的任务）。
- `silly_priming.py` 的句子故意荒谬，不适用通用性和语体一致性维度。
