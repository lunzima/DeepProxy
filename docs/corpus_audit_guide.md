# 提示工程语料审查指引（自包含执行版）

本文件用于让外部模型独立完成 DeepProxy 项目内全部提示工程语料的质量审查。所有规则、红线与审查要点均内嵌于本文件，不需要读取本仓库其他任何文档。

合并自 docs/prompt_sentence_review.md 与 docs/corpus_audit_guide.md。

## 一、审查任务

DeepProxy 是一个 LLM 代理服务。每次推理请求会向上游模型注入若干提示工程语料，分布在六个文件中。审查目标是识别仍存在质量问题的句子，提出改写建议，不要求重写整段或整篮。

| 序号 | 文件 | 数量 | 功能 | 注入方式 |
|---|---|---|---|---|
| 1 | `deep_proxy/optimization/skills_general.py` | 14 条多行字符串，约 100 句 | A 组通用风格、B 组反幻觉、C 组上下文技能、CoT 模板、日期注入 | 拼接后注入 system |
| 2 | `deep_proxy/optimization/skills_transform.py` | 1 条短语，1 条导入 | D 组消息转换：RE2 追加 user 消息尾部 | 直接追加到 user 消息 |
| 3 | `deep_proxy/optimization/silly_priming.py` | 20 条列表字符串 | 实验性无厘头断言扰动 | 注入 system 最前 |
| 4 | `deep_proxy/optimization/compressor.py` | 1 大段元提示，约 25 句 | LLM 压缩器指令 | 发给压缩 LLM，不入最终请求 |
| 5 | `deep_proxy/optimization/dynamic_baskets.py` | 3 套 × 3 篮 × 16 句 = 144 句 | 场景化短段注入 | 追加到 system 末尾 |
| 6 | `deep_proxy/compatibility/reasoning_handler.py` | 1 条多句占位，约 3 句 | dummy reasoning_content 回退 | 注入 assistant 消息 |

四个语料桶（属于上表第 3、5 项）抽样机制：

- 编程桶 / 创意桶 / 通用桶各 3 个篮子（methodology / best_practices / moderate_encouragement），每请求从每篮独立抽 1 句拼为一段，共 3 段
- 弱智吧桶每请求不放回随机抽 2 条，包装为带署名的引文形态

## 二、双重审查规则

每一句必须通过两层独立判定，任一层不通过即需修改。

### 2.1 第一层 句子内容是否教模型违规

句子作为指引，会让模型养成对应的写作或编码习惯。这种习惯本身必须符合下方红线。例：若一句教「以仿佛 X 一样为开头比喻」，则该句教模型违反陈腐表达红线，必须标记。

### 2.2 第二层 句子本身是否违规

句子作为中文文本，它的措辞自身也要符合下方红线。例：句子里出现「岁月」或破折号或「不是…而是…」句式，本身就违规。

## 三、红线清单

以下规则源自项目通用创作要求。本指引内嵌完整核心条款。

### R1 禁用破折号

叙事文本及方法论指引中禁止使用破折号符号「——」。需要解释、转折、递进、因果时使用逗号、句号或分号。

### R2 禁止否定对位

禁止「不是 A，而是 B」以及变体（「而非」等）的对位归因句式。改写为正向断言或并列陈述。

### R3 禁止陈腐表达

禁用以下及同类陈腐意象：

- 投入湖面的石子
- 指节泛白
- 心中一紧
- 五味杂陈
- 千头万绪
- 任何在网文或通俗写作中已被滥用至失去感染力的固定意象

### R4 避免文艺腔抽象词

作为情感载体禁用：岁月、时光、温柔、温暖、远方、梦想、灵魂、彼岸。情感须由具体的人物动作、环境变化、感官细节承载。

### R5 避免官僚腔

禁用公文动词：进行、予以、实施、落实、推进。禁用公文指代词：上述、该、本、其（在不自然语境中作指代用）。注意「本文」作为对当前文档的指代命中红线，应改为「正文」「此处」等。

### R6 避免学术腔

禁止用抽象名词替代具体描写。例「权力结构的系统性缺陷」应写具体的人与事。禁止动作名词化收尾，如「文本的展开」「结论的得出过程」「可被快速扫读」等结构属轻度学术腔，可接受但不宜密集。

### R7 避免文言腔

「之 / 乎 / 者 / 也」以及「各得其位」「言之凿凿」等仿古短语在中性语境中视为违规。

### R8 避免说明书腔

禁止条目式罗列与「按 X、Y、Z 三层逐层组织」型枚举骨架。建议改为连贯句。

### R9 避免「仿佛 / 似乎 / 好像」开头比喻

该开头模式在网文中已被滥用，禁用。

### R10 避免「了」短句堆砌

禁止「他放下茶杯了。看着窗外了。街上没人了」型短促陈述句连续出现。叙事中「了」应作长句内的体标记，而非短句结尾的语气词。

### R11 避免叙事者跳出来归因

针对创意桶尤其重要。禁止教模型用「本性 / 天性 / 宿命 / 本质 / 源于 / 骨子里」等词替读者完成因果推导。让行为、细节、对话本身传递信息。

### R12 句式同质化警戒

同篮 16 句中某一开头模板（如「先 X，让 Y」「可以为 X，让 Y」「X 之前先 Y」）出现 ≥ 60% 视为同质化警戒线。

## 四、通用审查 4 维度

R1 至 R12 红线之外，通用审查仍需覆盖以下 4 维度。维度 4「效能」仅适用于最终注入给 LLM 的提示句，对 compressor.py 元提示不适用。

### 维度 1 正确性（必须修）

- 语法 / 搭配：是否存在搭配不当或语法错误
- 歧义风险：句子能否被不同方式理解，是否会在模型侧造成误导
- 术语误用：词汇在目标语境中含义是否成立

### 维度 2 通用性（建议修）

- 场景覆盖：句子是否只适用于所属模块场景内的某个极端子集
- 自足性：句子脱离上下文能否独立理解，是否依赖未给出的前置知识
- 术语透明度：专业术语是否可被直接理解，还是需要额外解说

### 维度 3 语体一致性（建议修）

- 口语化警戒：是否存在过于口语的词汇，与同组其他句子语体脱节
- 书面化警戒：是否存在过度的文学或古典用词，在目标场景中显得不协调
- 组内一致性：同一功能组内各句子在语体上是否协调

### 维度 4 效能（可选修，仅最终提示句）

- 信息密度：句子是否足够精炼，能否用更少 token 表达相同含义
- 行动锚点：句子是否给模型提供了可实际执行的操作指引，还是停留在抽象口号
- 肯定句式：对于设计要求「仅肯定句」的模块（如 dynamic_baskets），逐句核实是否违反
- 冗余修饰：是否存在堆叠近义词或多层修饰却不增加语义的情况

## 五、三大风险维度

### V1 疑义

- 句子能否被两种合理方式理解
- 关键词指代是否清晰
- 句子脱离上下文能否独立理解

### V2 重复

- 同篮内重复：16 句内是否有近义指引
- 跨篮重复：同一指引是否在 methodology / best_practices / moderate_encouragement 三篮间复述
- 跨桶重复：编程桶 / 创意桶 / 通用桶之间是否有同一模板的迁移使用
- 术语堆叠：高频文学术语（分量 / 节奏 / 质地 / 呼吸 / 克制 / 厚度 / 肌理 / 锚点 / 伏脉 等）在 48 句内出现 ≥ 4 次视为堆叠，3 次为临界

### V3 偷懒借口

这是 moderate_encouragement 篮的核心风险。该篮的「可以 / 值得 / 允许」句必须严格审视：

- 「允许保留 X」是否会被模型读为「也允许不做 Y」
- 「可以为 X 多 Y」是否会被读为「也可以不做」
- 「留白 / 保留 / 不写出口 / 节制」类指引是否给模型留下「少写也合规」的合理化空间
- 「可以再 X 一次」软指令的「可以」是否使其退化为可选项

修复方向：把软指令转为肯定句、把鼓励词换为对动作的明确锚定（「在 X 时刻」「每隔几个回合」）、把模糊数量词（「略多」「一两句」「稍宽裕」）换为可校验的边界（「比最低需要略多」「把 X 写充分」）。

## 六、各文件专用审查要点

### 6.1 skills_general.py

格式：14 条独立的长多行字符串，每条覆盖一个完整的功能主题。

附加要点：

- 核心指令与例证边界：条目的主体部分（祈使指令）和示例 / 注释部分是否清晰可辨。模型是否会误将例证当作指令
- 条件守卫：有些技能（如 `_SKILL_SHOW_MATH_STEPS`）含「仅当…才…」的条件逻辑。条件是否足够精确，不会因过宽导致不当触发或过窄从不触发
- 价值观叠加：安全 / 反谄媚约束是否过度强化了某种偏向。是否因此在非安全话题上也引入了不必要的社会偏好
- 过时引用：`_SKILL_AVOID_FABRICATED_CITATIONS` 和 `_SKILL_REASON_GENUINELY` 等条目提到了具体术语或 AI 行业现象。这些引用是否仍然反映当前的模型行为特征
- 多祈使句链：含多个祈使句的条目中，句子之间的优先级是否明确。是否存在隐含的条件冲突（如「完全基于自身推理」与「从多方面收集信息」）

### 6.2 skills_transform.py

格式：1 条 6 字短语 + 1 个从 skills_general.py 导入的 CoT 提示。

附加要点：

- `_RE2_MARKER` 形如「\n再读一遍这个问题：」，后接复制的 user 消息原文。拼接后的整体是否自然，是否可能造成模型困惑
- CoT system prompt（从 skills_general 导入）按 skills_general 的审查标准处理即可

### 6.3 silly_priming.py

格式：20 条短列表字符串。这些句子故意荒谬或不合逻辑，用于扰动 MoE 路由器。通用性和语体一致性维度不适用。

反向锚定：修改 `silly_priming.py` 池内容或调整注入条数前，请同步检查该文件 docstring 顶部的「审查豁免」段。该段定义了曝光率阈值与扩池或提注入率的安全方向区分。维护需保持本节与 docstring 双向一致。

不适用维度：通用性、语体一致性、文学化警戒、口语化警戒、不雅措辞、不合常理。AMD 原典的不雅措辞为文化典故必须保留。

仅核查：

- 中文语法是否成立（荒谬不等于病句）
- 低语义污染（核心红线）：是否无意中形成可执行指令。例如「复制三遍」「按 X 输出」等会污染真实任务的指令性内容必须移除
- 句间是否重复同一机制 + 同一深层语义模板
- 句式同质度：「X 以后/之后/后」等开头模式 > 30% 视为偏高
- 偷懒借口：「X 不需要做 Y / 可以省略」类放任型断言

已知设计参数：

- 池规模 20，每请求注入 2 条，单条曝光率 10%（≤30% 视为低风险）
- 注入位置：system 消息最前
- 注入形态：每条搭配中性出处署名形成引文块

### 6.4 compressor.py

格式：1 大段元提示，发给压缩专用 LLM 的指令。不适用效能维度，因为信息密度与冗余修饰是它自身要做的任务。

附加要点：

- 指令完备性：是否涵盖了所有需要保护的元素（保留规则 / 命名实体 / 带引号字符串 / 代码 / JSON 字段 / XML 标签）。是否有未提及但可能被压缩 LLM 意外删除的关键元素
- 约束可操作性：约束条目（「不得删弱规则」「不得合并不同情形」「不得否定翻转」）是否足够具体，压缩 LLM 能否可靠遵循
- 优先级一致性：多个约束之间是否存在隐含冲突（如「压缩到 40-60% tokens」与「逐字保留所有带引号字符串」在极端情况下可能矛盾）
- 输出格式：是否清晰指定了只输出改写后文本、无额外内容。输出格式的约束是否存在歧义

### 6.5 dynamic_baskets.py 总要点

格式：3 套 × 3 篮 × 16 句 = 144 句嵌套字典。按通用审查 4 维度逐句审查即可。

creative 套额外受项目通用创作要求 12 条红线约束（详见对应规约段）。

### 6.6 编程桶 `_CODING_BASKETS`

特殊关注：

- 过度抽象：是否教模型违反 YAGNI（「为未来扩展预留接口」类指引应只针对已确认场景）
- 过度防御：是否教模型对内部代码做不必要的边界 / 异常 / 验证（红线只允许在系统边界做防御）
- 无谓注释：是否教模型为对接外部依赖、设计接口前先写一段说明
- 回答注水：是否教模型在响应开头复述用户需求。该动作应作为内部对齐，不要对外复述
- 学术词套用：「守恒约束 / 不变量 / 可解释性」等抽象名词若超出业务实际语境则违规

句式同质化阈值：methodology 篮「先 X，让 Y」模板 ≥ 11/16，moderate_encouragement 篮「可以 / 值得 / 允许」模板 ≥ 12/16 视为锁死。

### 6.7 创意写作桶 `_CREATIVE_BASKETS`

红线全部适用，特别严格执行 R3、R4、R9、R11。

特殊关注：

- 句子内容是否教模型在叙事中「不是…而是…」做因果归纳（R11 模式 A）
- 句子内容是否教模型让叙事者跳出来解释（违反「展现而非讲述」原则）
- 鼓励「留白 / 保留 / 不写出口 / 克制」类指引是否被读为「少写也行」（V3 偷懒借口）
- 术语堆叠：分量 / 节奏 / 质地 / 呼吸 / 克制 / 厚度 / 肌理 / 锚点 / 伏脉 在 48 句中各项超过 3 次为堆叠
- 抉择 / 行动 / 心理描写的关系：教「留给人物自身去走」必须明确「由后续情节承接」而不能让模型省略此处

LLM 可执行性：本桶句子是给 LLM 读的，因此「朗读 / 读出声 / 听觉」等指令模型无法执行，应改为「默念 / 在脑内预演语势」。

### 6.8 通用写作桶 `_GENERAL_BASKETS`

红线适用，但 R3、R4、R11 文艺与小说叙事相关条款的适用性较弱。

特殊关注：

- 「本文」作为指代命中 R5 官僚指代红线，必修
- 重复族群：过渡 / 衔接、术语 / 概念解释、数据 / 数字、读者 / 摘要等高频名词族在 48 句内反复使用同一指引时构成 V2 重复
- 数字锚点重复：「三到五个要点」与「三到五句摘要」两处雷同时建议二选一去数字
- 条目式枚举骨架（「按事实、判断、建议三层逐层组织」）违反 R8，应改为连贯句

### 6.9 reasoning_handler.py

格式：1 条多句占位文字。审查时无需逐句分解。

附加要点：

- 自洽性：当 LLM 看到这句 dummy reasoning 被回传时，是否会将其误解为上一轮的完整推理。需要注意其措辞是「请对当前请求重新思考」，方向是否正确
- 语言中立性：占位文字是否为中文（与项目一致），是否不含 time-sensitive 引用（如具体日期）
- 信号完整性：占位文字是否提示了模型「之前推理可能不正确需要重新验证」。这个信号是否够强以避免模型跳过推理，是否过强以至于模型过度怀疑自身的推理

## 七、注释维护要求

### 7.1 弱智吧桶注释体例

每条仅保留单行机制标签（如「# 因果倒置」「# 类别错位」「# 谐音双关」）。禁止重新引入以下旧体例：

- 「Better-Ruozhiba 原帖：…」（出处引用）
- 「修订 A：…」（人工驳斥版本）
- 「人工 A 字面化驳斥…」（元解释）

机制标签举例：参考系混淆 / 因果倒置 / 字面化成语 / 类别错位 / 成分反转 / 谐音双关 / 谐音拆解 / 强行因果 / 拟人 + 双关 / 自指悖谬 / 局部反逻辑 / 反逻辑画面 / 守恒违反 / 自我中心错觉 / 范围谬误 / 物质同一性误推 / 同音双关 / 价值范畴越界 / 量纲错位。

### 7.2 dynamic_baskets.py 注释要求

- 模块顶部 docstring 保留：结构 / 抽样 / 注入位置 / 规则 / 场景 / 空篮容错 6 条要点
- 每个 `_XXX_BASKETS` 字典前保留一行 section header 注释，标注其用途与采样 profile 映射
- 篮内单句禁止附加 inline 注释（机制标签 / 参考来源 / 修订原因等均不应保留）

### 7.3 silly_priming.py 注释要求

- 模块顶部 docstring 保留：设计参考（学术依据） + 注入约束（操作依据） + 审查豁免（精简版，仅含豁免范围 / 复查规则两点 / 安全边际数字 / 调用点指引）
- 池前一段保留分类说明（AMD 典故 + 弱智吧体）
- 每条仅保留单行机制标签（按 7.1 体例）
- 包装层注释保留必要的设计动机一句话，不需要多层 bullet 嵌套

### 7.4 整体注释原则

删除：双向锚定 meta-rule、调整方向区分的 meta-meta-rule、个别条目的来源出处考证、超过 2 层嵌套 bullet、自我引用类（「详见本段 / 见上文 / 同下」等）。

保留：load-bearing 的安全数字阈值、调用点指引、操作约束。

## 九、审查规则细则

1. 每个问题给出：行号 + 原文 + 问题机制 + 改写建议。机制描述要具体（例「R5 官僚指代」或「V3 偷懒借口」），不要笼统说「措辞不佳」。
2. 必修 / 建议修 / 可选修 三档：
   - 必修：硬命中红线（破折号、否定对位、文艺抽象词、陈腐表达、教模型违规）、严重偷懒借口、可执行性问题
   - 建议修：句式同质化、术语堆叠超阈值、轻度学术腔、跨篮重复
   - 可选修：句尾词重复、近义词偏好、轻微冗余
3. 不要凑数：完全可接受的轻微差异（如「里」与「中」互换、「在 X 中」与「在 X 里」）不要列。
4. 不要重复已修过的判定：如果一条句子读起来很好，就明确说「无新问题」。
5. 改写建议必须是完整句：保持 16 句的篮子结构，每条建议改写后仍为肯定陈述句、句末以「。」结尾、与同篮其他句子语体协调。
6. 改写建议必须自身合规：建议的新句也要通过双重审查，不要在改写中引入新的红线违反。
